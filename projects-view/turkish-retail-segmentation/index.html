Overview

What this project is about

Retail businesses generate mountains of transactional data, but without structure, that data doesn't tell you much. This project takes a real Turkish retail dataset — covering customer demographics, purchase categories, payment methods, and spending patterns — and turns it into something you can actually reason from.

The goal was to segment customers meaningfully: not just by age or gender, but by how they actually behave as shoppers. Which groups spend the most? How does payment preference differ across age brackets? What does the gender split look like across product categories? These are the kinds of questions this project was built to answer.

Phase 01

SQL Analysis in MySQL

Before any dashboard, the data had to be interrogated at the query level. Using MySQL, I worked through nine structured analytical queries — each one designed to pull a specific insight out of the dataset. The queries progressively built up a picture of customer behavior, from simple aggregations to more nuanced breakdowns.

One thing that mattered a lot here was being precise about what was actually being calculated. For example, there's a difference between using unit price and using the total transaction value — and getting that wrong would've skewed the spending analysis entirely. That kind of attention to what the data actually represents, not just what the column is named, shaped how every query was written.

Phase 02

Power BI Dashboard

With the SQL analysis complete, the next step was making those insights visual and interactive. The Power BI dashboard was built with a direct ODBC connection to MySQL — meaning the data flows live from the database, not a static export. DAX measures were written to handle custom calculations for the KPIs displayed across the report.

The dashboard is split across two focused pages, each tackling a different lens of the customer data. Rather than trying to pack everything into a single screen, each page has a specific analytical angle — which keeps the visuals clean and the story easy to follow.

Insights

What the data revealed

The analysis surfaced some genuinely useful patterns — the kind that would matter for a real retail business thinking about targeting, inventory, or payment infrastructure.

Spending by Gender

Female customers showed higher average transaction values in certain categories, while male customers had more volume concentrated in a narrower set of product types.

Age-Based Patterns

Middle-aged segments (30–50) drove the bulk of revenue. Younger segments showed preference for specific categories, suggesting different targeting strategies could be effective.

Payment Preferences

Payment method usage wasn't uniform across age groups. Certain methods were heavily skewed toward older customers, while others were more evenly distributed — a pattern that has implications for checkout experience design.

Category Concentration

Revenue was not spread evenly across product categories. A small number of categories accounted for a disproportionately large share of total sales, highlighting where the business has its strongest demand signal.

High-Value Customers

A subset of customers consistently spent well above the average across multiple transactions. Identifying this group is the first step toward any retention or loyalty-based strategy.

Cross-Segment Behavior

When gender and payment method were analyzed together, distinct behavioral clusters became visible — groups that don't emerge when looking at each variable in isolation.

Reflection

What this project taught me

🔍
Data precision matters more than you think
There were moments where a single column choice — unit price versus total transaction value — would have produced completely different results. Learning to read what data actually represents before writing queries is something that only comes through doing the work.
🔗
Connecting the pipeline end-to-end
Setting up a live ODBC connection from MySQL to Power BI was a real integration exercise. It meant understanding how both tools talk to each other, not just using them independently. That kind of end-to-end thinking changes how you approach building a report.
📊
Visuals only land if the structure is right
The dashboard only worked well because the SQL layer was solid. If the queries were wrong, the visuals would've looked fine but told a false story. Getting the data model right before touching Power BI was the right order of operations.
